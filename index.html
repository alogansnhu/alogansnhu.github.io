Professional Self-Assessment

The self-assessment is presented first because it frames the entire portfolio.

The capstone required more than collecting artifacts. It required me to explain how coursework across the computer science program changed the way I design systems, communicate decisions, and apply security-minded thinking to real code.

Professional growth

From isolated assignments to professional software decisions

Completing the Computer Science program and building this ePortfolio helped me move from writing code that simply works to building software that is easier to maintain, test, explain, and secure. The capstone made that growth visible because I had to return to earlier artifacts, identify their limits honestly, and strengthen them without losing their original purpose. That process changed how I think about design trade-offs. I now pay closer attention to separation of concerns, validation, evidence for correctness, and the value that each technical decision adds for the intended audience.

Beyond the artifacts included in this portfolio, milestone-based coursework in software lifecycle planning and software testing taught me to break large requirements into manageable increments, define acceptance evidence before claiming a feature is complete, and revise work based on structured feedback. Those habits are now part of how I approach every project.

Collaboration and communication

Working as if the next reader matters

One of the most important lessons from the program is that computer science work is collaborative even when an assignment begins as individual work. The code review portion of this capstone required me to explain existing functionality, identify risks, and justify enhancement plans in a format that peers, instructors, or managers can use to support decision-making. That same mindset shows up in my source code. Clear naming, modular responsibilities, reusable validation, and written narratives all make handoff and review easier.

The program also strengthened how I communicate with stakeholders. For technical audiences, I can point to implementation details such as scheduling indexes, MongoDB aggregation pipelines, validation utilities, and audit logging. For non-technical audiences, I can translate those details into benefits such as safer data handling, cleaner architecture, more predictable behavior, and stronger decision-support tools.

Technical strengths

Data structures, software engineering, databases, and security

The strongest technical theme in my coursework is backend and data-driven development. In data structures and algorithms work, I learned that structure selection should reflect the behavior a system needs to support, not just the easiest collection to use. That is why the enhanced Java artifact now relies on keyed lookup, priority ordering, chronological indexing, and conflict detection instead of only simple identifier-based CRUD.

In software engineering and database work, I learned to treat architecture and data access as connected concerns. Reusable abstractions, cleaner module boundaries, and database-side analytics create systems that are easier to extend and easier to explain. Security also became a design habit rather than an afterthought. Removing hardcoded credentials, validating inputs, restricting unsafe operations, and preventing invalid states are all practices that now shape the way I write code.

Portfolio synthesis

How the artifacts fit together

The artifacts in this portfolio show one consistent direction: building maintainable systems that combine structured data, thoughtful service design, and clear communication. The code review establishes the evaluation process behind the work. The software engineering and database enhancements show how a MongoDB-backed dashboard can be transformed into a more modular and secure application. The algorithms and data structures enhancement shows how a basic Java service suite can evolve into a scheduling-oriented system with stronger retrieval behavior, conflict management, and validation.

Taken together, these artifacts represent the kind of developer I aim to be: deliberate about design trade-offs, careful with testing and security, and able to explain technical decisions clearly to different audiences.

Category One

Software engineering and design artifact

For the software engineering and design category, I used the CS 340 animal shelter dashboard artifact and focused the enhancement on modularity, configuration, responsibility boundaries, and maintainable backend structure.

Original course: CS 340 Original artifact: ProjectTwoDashboard.ipynb Enhanced artifact: modular Dash + Python bundle
Narrative summary

Why this artifact belongs in the portfolio

The original artifact demonstrated practical database-backed functionality, but too much responsibility was concentrated in the dashboard layer. The enhancement reworked the project into a more deliberate software design by separating configuration, CRUD behavior, rescue query logic, analytics, and presentation responsibilities.

The most important software engineering improvements were moving credentials into environment-based configuration, introducing validated CRUD operations with controlled projection, sort, and limit behavior, and extracting rescue-specific query logic into its own module. Those changes made the artifact easier to reason about, easier to extend, and more representative of production-style design choices.

Category Two

Algorithms and data structures artifact

For the algorithms and data structures category, I used the CS 320 service suite and enhanced it into a more realistic scheduling system with priority-aware task retrieval, chronological appointment indexing, and conflict detection.

Original course: CS 320 Original artifact: module 6 contact, task, and appointment services Enhanced artifact: scheduling-oriented Java bundle
Narrative summary

How the enhancement changed the artifact

The original service suite already showed validation and service-level CRUD behavior, but the three services were still largely independent in-memory classes built around basic HashMap storage. The original task service focused on ID-based create, update, and delete operations, while the original appointment service only enforced unique IDs and basic lookup and deletion. The enhancement redesigned the internal organization of tasks and appointments so the program could support more useful retrieval and more deliberate correctness checks.

Tasks now track both priority and due dates, allowing the service to return urgent work first instead of only supporting direct ID lookup. Appointments now use a chronological index by day and reject overlapping time slots. The artifact also gained shared CRUD abstractions, reusable validation utilities, a facade, and audit logging so the solution feels like a coherent service layer rather than a set of unrelated classes.

Original artifact Enhanced artifact
HashMap-backed task CRUD with ID lookup only Priority ordering and deadline-based task retrieval
Appointment add, delete, and lookup without scheduling logic Chronological day buckets with overlap detection
Three standalone service classes with no shared abstraction layer Shared CRUD abstractions, facade coordination, and audit trail support
Validation focused on field rules and unique identifiers Validation extended into scheduling correctness and safer update paths
Key evidence

Data structures chosen for behavior, not convenience

  • TreeSet-backed task ordering for urgency-based retrieval
  • TreeMap<LocalDate, List<Appointment>> for day-based scheduling and date-range queries
  • Binary-search insertion plus adjacency conflict checks for appointments
  • Common abstractions and audit logging that support maintainability around the algorithmic changes

Category Three

Databases artifact

For the database category, I returned to the CS 340 animal shelter project and emphasized stronger query control, MongoDB aggregation, and data-access decisions that support analytics and safer database interaction.

Narrative summary

Why the database enhancement is distinct

The same CS 340 artifact works for both software engineering and database outcomes because the final enhancement changed not only the application structure, but also the way database work is performed. Instead of leaving the project at the level of basic CRUD, the enhancement adds validated projections, controlled sort and limit behavior, aggregation pipelines for summary analytics, and geolocation-oriented retrieval support.

The database layer also now rejects unsafe field names and restricts which query, update, and aggregation operators are allowed. That change demonstrates a stronger security mindset because it treats database access as an interface that must be validated, not just a thin pass-through to MongoDB.

Database-oriented improvements

  • Geolocation point retrieval for map-driven dashboard views
  • Sanitized query, projection, sort, update, and aggregation stage handling
  • Environment-based credentials instead of hardcoded database secrets
